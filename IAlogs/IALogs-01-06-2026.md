### ✅ Sessão 1 — Estrutura base e dados sintéticos

**Contexto:**  
Banco de dados real (scraping Sofascore) ainda não disponível neste dispositivo.
Decisão: usar dados sintéticos para desenvolvimento até migração do banco.

**Realizado:**
- Definida estrutura de diretórios do projeto (`data/raw`, `data/processed`, `agent/notebooks`, `agent/src`, `agent/models`)
- Criado `generate_data.py` — gerador de dados sintéticos com distribuições realistas
  - 3 ligas: Brasileirão Série A, Premier League, La Liga
  - 5 temporadas por liga (20/21 → 24/25)
  - ~5.700 partidas, ~38 estatísticas por partida (xG, posse, chutes, cartões, escanteios, passes, duelos, etc.)
- Gerados `matches.csv` (5.700 linhas) e `match_stats.csv` (5.400 linhas)

### ✅ Sessão 2 — Notebooks de análise e treinamento

**Notebook 01 — Exploração dos dados:**
- Análise de cobertura temporal, distribuição de resultados e correlações
- Resultado: distribuição realista (H≈46%, D≈25%, A≈29%)
- Cobertura de stats: ~95% dos jogos

**Notebook 02 — Preprocessamento e Feature Engineering:**
- Rolling window de 10 jogos por time (sem leakage temporal)
- ~71 features geradas: aproveitamento, gols, xG, posse, passes, chutes, escanteios, cartões, duelos aéreos, H2H, posição normalizada na tabela, médias da liga
- Split temporal: treino < 01/07/2023, validação >= 01/07/2023
- Exportados `features.csv` (5.474 × 77) e `targets.csv` (5.474 × 6)

**Decisão técnica — Over/Under:**  
Targets de cartões e escanteios migrados de 3 classes (baixo/médio/alto) para binário Over/Under:
- Cartões: Over/Under 6.5 amarelos totais
- Escanteios: Over/Under 10.5 escanteios totais  
*Motivo: alinhamento com mercado real de apostas, mais interpretável.*

**Notebook 03 — Treinamento da rede neural:**
- Arquitetura: backbone `Dense(256→128→64)` + 3 cabeças independentes
  - Resultado: `Softmax(3)` — H / D / A
  - Cartões: `Softmax(2)` — Over/Under 6.5
  - Escanteios: `Softmax(2)` — Over/Under 10.5
- Early stopping na época 25, melhor val_loss: 2.7914
- Métricas de validação:

| Alvo | Acurácia | Baseline |
|---|---|---|
| Resultado | 0.4062 | 0.333 |
| Cartões (O/U 6.5) | 0.4038 | 0.500 |
| Escanteios (O/U 10.5) | 0.4869 | 0.500 |

**Notebook 04 — Testes do agente:**
- 5 cenários testados com sucesso (Flamengo×Grêmio, Man City×Leeds, Real Madrid×Getafe, Gre-Nal, Liverpool×Man City)

### ✅ Sessão 3 — Predictor e Tools

**Arquivos criados:**
- `data_loader.py` — cache em memória para os CSVs, ponto único de acesso
- `predictor.py` — carrega o `.pt`, expõe `predict(home, away, league)`
- `tools.py` — 5 tools com schemas para Claude API / Ollama tool use:
  - `get_team_stats`, `get_h2h`, `get_league_table`, `get_team_schedule`, `run_prediction_engine`
- `test_tools.py` — smoke test de todas as tools sem LLM

**Resultado dos testes:**  
Tools 1–4 funcionaram perfeitamente. Tool 5 (`run_prediction_engine`) funcionou após correção da arquitetura no `predictor.py` (heads binários).

### ✅ Sessão 4 — LLM Client e Orchestrator

**Provider escolhido:** Ollama Cloud (`https://ollama.com`)  
**Modelo:** `gpt-oss:120b`  
*Motivo: requisito escolar de uso do Ollama; modelo cloud evita sobrecarga local.*

**Arquivos criados:**
- `.env.example` — template de configuração
- `llm_client.py` — wrapper Ollama Cloud com suporte a tool use, converte schemas Claude API → formato Ollama/OpenAI
- `orchestrator.py` — agentic loop com até 5 rounds de tool calling, system prompt em português, modo terminal interativo

**Resultado:**  
Agente respondendo corretamente em linguagem natural. Ajuste de tom aplicado no system prompt: estilo conversacional sem negritos ou seções formatadas.

**Ambiente:**
- macOS, conda env `oscabet` (Python 3.11)
- Ollama instalado via Homebrew
- Biblioteca `ollama` instalada via pip no env
